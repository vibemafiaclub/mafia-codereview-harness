# 문서 업데이트 (code-convention / ADR)

`docs/code-convention.yaml` 또는 `docs/adr.yaml`에 항목을 추가/수정/삭제한다.

## 사용법

```
/update-docs convention  — code-convention 항목 추가/수정/삭제
/update-docs adr         — ADR 항목 추가/수정/삭제
```

인자 없이 호출하면 두 문서 모두를 업데이트한다.

## 공통: 컨텍스트 분석 우선 원칙

어떤 문서를 업데이트하든, 지금까지의 작업 내용을 먼저 읽고 분석한다. **기존 문서 전체도 먼저 읽고 분석**해야한다.

1. `docs/code-convention.yaml`과 `docs/adr.yaml`을 **모두** 읽는다.
2. 현재 작업의 의사결정과 기존 항목을 대조하여 다음을 판단한다:
   - **충돌 항목**: 현재 작업에서의 결정(또는 새로 추가할 항목)이 기존 항목과 모순되는가?
     - 예: 새 ADR이 "에러를 중앙에서 처리"로 결정했는데, convention에 "에러는 발생 지점에서 처리"가 남아 있는 경우
   - **연쇄 수정**: 한 문서의 변경이 다른 문서의 항목에도 영향을 미치는가?
     - 예: ADR에서 새 기술 결정을 채택하면, 해당 결정과 반대되는 convention 규칙도 수정/삭제가 필요
   - **폐기 대상**: 기존 항목 중 이번 변경으로 인해 더 이상 유효하지 않은 것이 있는가?
3. 분석 결과 추가/수정/삭제해야할 항목들을 사용자에게 **변경 제안 목록**으로 정리하여 보여준다:
   - MUST / RECOMMENDED 로 등급을 나누어 제안한다.
4. 사용자가 최종 결정한 내용을 반영한다.

## code-convention 업데이트 절차

1. `docs/code-convention.yaml`을 읽는다.
2. `docs/adr.yaml`도 함께 읽고, 현재 작업에서의 결정이 기존 ADR 결정과 충돌하지 않는지 확인한다.
3. 분석 결과 추가/수정/삭제해야할 항목들을 사용자에게 **변경 제안 목록**으로 정리하여 보여준다:
   - MUST / RECOMMENDED 로 등급을 나누어 제안한다.
4. 사용자의 요청에 따라 항목을 추가/수정/삭제한다.
5. 항목 작성 시 다음 스키마를 따른다:
   ```yaml
   - id: {카테고리}-{번호}    # GEN, TS, NEXT, NEST, MONO 등
     rule: ...                 # 명확하고 실행 가능한 규칙 서술
     stacks: [...]             # 적용 대상 스택. 전체 적용이면 [all]
   ```
6. 변경 내용을 사용자에게 알린다.

## ADR 업데이트 절차

1. `docs/adr.yaml`을 읽는다.
2. `docs/code-convention.yaml`도 함께 읽고, 새로운 ADR 결정과 모순되는 convention 규칙이 있는지 확인한다.
3. 분석 결과 추가/수정/삭제해야할 항목들을 사용자에게 **변경 제안 목록**으로 정리하여 보여준다:
   - MUST / RECOMMENDED 로 등급을 나누어 제안한다.
4. 항목 작성 시 다음 스키마를 따른다:
   ```yaml
   - id: ADR-{번호}
     title: ...
     status: adopted | deprecated | superseded
     date: YYYY-MM-DD
     stacks: [...]
     context: |
       (구체적인 상황 서술 — 아래 가이드라인 참조)
     decision: ...
     consequence: ...
   ```
5. **context 작성 가이드라인:**
   - context만 읽고도 "왜 이 결정이 필요했는가"를 구체적으로 연상할 수 있어야 한다.
   - 나쁜 예: "에러 처리가 산발적이라 비일관적"
   - 좋은 예: "각 서비스 레이어에서 에러를 자체적으로 catch하고 임의 형식으로 반환하고 있었다. 어떤 엔드포인트는 { message: "..." }를, 다른 엔드포인트는 { error: "...", code: 500 }을 반환했다. 프론트엔드에서 API 에러를 처리하려면 엔드포인트별로 에러 형식을 확인해야 했다."
   - 당시 어떤 문제가 있었는지, 누가/어디서 고통받았는지, 어떤 시도가 실패했는지를 포함한다.
6. context가 불충분하다고 판단되면 사용자에게 이유와 함께 구체화 방안 논의를 요청한다.
7. 변경 내용을 사용자에게 확인받는다.
